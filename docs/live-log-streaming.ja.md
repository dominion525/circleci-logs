# ライブログストリーミング

[English version](live-log-streaming.md)

## 背景

CircleCI のステップがまだ実行中の場合、ログを一括取得しても「その時点までの出力」しか見えない。デプロイや結合テスト、Docker ビルドなど長時間かかるステップでは、新しい出力を見るために何度もコマンドを再実行し、ログ全体を読み直す必要がある。

ライブログストリーミングは CircleCI の非公開出力 API をインクリメンタルにポーリングし、新しいバイトが到着するたびに表示することでこの問題を解決する。リモート CI ログに対する `tail -f` のような体験を提供する。

## 動作の全体像

```
show_log()
  │
  ├── action.status が "running"？
  │     ├── はい ─→ stream_log()    （ライブストリーミングループ）
  │     └── いいえ ─→ ログ一括取得  （従来と同じワンショット）
  │
  stream_log()
  │
  ├── ヘッダー表示（ワークフロー、ジョブ、ステップ、"streaming..."）
  ├── raw mode 有効化（RawModeGuard）
  │
  └── loop
        ├── event::poll(50ms)         ← Esc / q / Ctrl+C チェック
        ├── fetch_private_output_range(byte_offset)
        │     └── vt100 parser → rows_formatted → 新規行のみ出力
        ├── 5 ポーリングごと: ジョブ詳細を再取得
        │     └── is_step_finished()?
        │           ├── はい ─→ 最終フェッチ → ステータス表示 → break
        │           └── いいえ ─→ 続行
        └── sleep(poll_interval_ms)   ← 通常 950ms
```

## インクリメンタルフェッチ

各ポーリングで `fetch_private_output_range` を呼び出し、HTTP `Range` ヘッダーで現在の `byte_offset` 以降のバイトのみを要求する。

| HTTP ステータス | 意味                               | `new_offset` の計算                     |
|-----------------|------------------------------------|-----------------------------------------|
| 200             | 全ボディ（Range 無視/初回）        | `body.len()`                             |
| 206             | 部分コンテンツ                     | `byte_offset + body.len()`               |
| 204             | 新しい出力なし                     | 変更なし                                 |
| 416             | Range 不正                         | 変更なし                                 |

### vt100 増分レンダリング

ストリーミングでは非常に大きな仮想スクリーン（`u16::MAX` 行）を持つ `vt100::Parser` を使って受信バイトを処理する。各チャンクを `parser.process(data)` でパーサーに投入した後、`render_vt100_rows` が `last_printed_row` から `cursor_position().0` までの新規行のみを `rows_formatted()` で取得し、実端末に出力する。

仮想スクリーンが事実上無限の行数を持つため、コンテンツが上方にスクロールアウトすることはない。`last_printed_row` トラッカーにより各行は実端末に正確に1回だけ出力される。出力は端末の通常のスクロールバックバッファに直接書き込まれるため、ユーザーはストリーミング中にスクロールアップできる（`tail -f` と同様）。

このアプローチにより、Docker BuildKit 等のツールが使用するカーソル制御シーケンス（cursor-up、erase-line、キャリッジリターンベースのプログレスバー）を正しく処理できる。vt100 レンダリングなしでは、これらのシーケンスが生のまま通過し出力が崩壊する（例: 91 行であるべき出力が 14,000 行以上に膨れ上がる）。

端末の幅は `crossterm::terminal::size()` で取得し（取得失敗時は `(80, 24)` にフォールバック）、仮想スクリーンの列数が実端末に一致するようにする。

NUL バイト（`\x00`）は一部の CI ログデータに含まれるため、入力時（パーサーへの投入前）と出力時（`rows_formatted` の結果から）の両方でフィルタリングする。

## ポーリング戦略

| タイマー         | 値       | 根拠                                                              |
|------------------|----------|-------------------------------------------------------------------|
| キー入力ポーリング | 50 ms   | 人間が知覚できる遅延（〜100 ms）より短い。キー操作が即座に反応する。 |
| フェッチ間隔     | 950 ms   | 50 ms + 950 ms ≈ 1 秒/サイクル。リアルタイム性と API 負荷のバランス。 |
| ステータスチェック | 5 ポーリング | 約 5 秒。ジョブ完了の検知遅延を許容範囲に収める。                  |

## レート制限

非公開出力 API はレート制限を適用しており、超過時に HTTP 429 を返す。`fetch_private_output_range` は `Retry-After` ヘッダーに従って最大 3 回リトライする（ヘッダーがない場合はデフォルト 1 秒）。

`stream_log` 内では一時的なフェッチエラーに対して適応型バックオフを適用する：

1. 初回エラー: `poll_interval_ms` が 950 ms → 3,000 ms に増加
2. 以降のエラー: 5,000 ms に増加
3. 連続エラーが 30 秒間続いた場合: メッセージを表示してストリーミング停止

成功したフェッチが発生すると、間隔とエラータイマーがリセットされる。

## Raw mode

### raw mode が行うこと

`crossterm::terminal::enable_raw_mode()` はターミナルドライバを再構成する：

- 行バッファリングの無効化 — キー入力が即座に配信される
- ローカルエコーの無効化 — 入力文字が表示されない
- Ctrl+C のシグナル処理の無効化（手動でハンドリングする）
- **LF → CRLF 自動変換の無効化**

### RawModeGuard（RAII パターン）

`RawModeGuard` は `enable_raw_mode()` / `disable_raw_mode()` をコンストラクタ / `Drop` のペアでラップする。これにより、ストリーミングループが `?` やエラー、パニックで終了した場合でもターミナルが復元される — すべての終了ポイントで手動クリーンアップを書く必要がない。

## エッジケース

| 条件                              | 動作                                             |
|-----------------------------------|--------------------------------------------------|
| ストリーミング中にステップ完了     | 最終フェッチ → ステータス行を表示 → ループ終了   |
| 連続フェッチエラーが 30 秒継続    | エラーメッセージを表示 → ループ終了               |
| Esc または q を押下               | ストリーム停止 → ログ後メニュー表示               |
| Ctrl+C を押下                     | ストリーム停止 → プログラム即終了                 |
| `build_num` が `None`             | ジョブ番号 0 にフォールバック（防御的処理）       |
| ステップが既に完了済み            | `show_log` がワンショットパスを使用（ストリーミングしない） |

## キーバインド

| キー     | アクション                                            |
|----------|-------------------------------------------------------|
| Esc      | ストリーミング停止 → ログ後メニュー（戻る / 終了）    |
| q        | Esc と同じ                                            |
| Ctrl+C   | ストリーミング停止 → プログラム終了                   |
