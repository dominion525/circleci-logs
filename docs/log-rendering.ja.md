# ログレンダリング

## 背景

CircleCI のジョブログにはターミナルエスケープシーケンス（ANSI カラーコードやカーソル制御シーケンス）が含まれている。これらは Docker buildx の進捗表示やカラー付きテスト出力など、リッチなターミナル出力を生成する CI ツールに由来する。

未処理のまま出力すると以下の問題が発生する：

- ファイルリダイレクト時にエスケープシーケンスがそのまま残る
- JSON 出力にエスケープシーケンスが混入する
- カーソル制御シーケンス（`\e[1A`）が CLI 自身のヘッダ表示を上書きするリスクがある

## CircleCI ログに含まれるエスケープシーケンスの種類

| 種類             | 例               | 用途                          |
|------------------|------------------|-------------------------------|
| SGR（色）        | `\e[34m`, `\e[0m`| 青文字、属性リセット          |
| カーソル上移動   | `\e[1A`          | カーソルを N 行上に移動       |
| 行消去           | `\e[2K`          | 現在行を全消去                |
| カーソル列移動   | `\e[0G`          | カーソルを 0 列目に移動       |
| カーソル表示制御 | `\e[?25l/h`      | カーソルの非表示/表示         |

Docker buildx はカーソル上移動＋行消去を使ってアニメーション的な進捗表示を実現している。ターミナル上では更新され続ける1つのブロックに見えるが、生バイト列としては数千回の上書きサイクルが記録される。

## データソース

`output_url`（v1.1 API の S3 署名 URL）と非公開 raw output API の両方がエスケープシーケンスを含むデータを返す。違いはサンプリング粒度のみ：

- **Private API**: ターミナルに書き込まれた生バイトストリーム — すべての更新が記録される（例: Docker ビルドステップで 14,000 行）
- **output_url**: 同じストリームをチャンク単位でバッチ化 — 中間状態が少ない（例: 同ステップで 4,300 行）

どちらも最終的なレンダリング結果は同一になる。

## レンダリング方式

[vt100](https://crates.io/crates/vt100) クレートでターミナルをエミュレートする。パーサーが生バイトストリームを処理し、すべてのカーソル移動を解決して最終的な画面状態を生成する。これは CircleCI Web UI の表示と同等の結果になる。

```rust
let mut parser = vt100::Parser::new(u16::MAX, 200, 0);
parser.process(raw_bytes);

// プレーンテキスト（エスケープシーケンスなし）
let plain = parser.screen().contents();

// ANSI カラーコード付きテキスト
let colored = parser.screen().contents_formatted();
```

設計上のポイント: ターミナルの高さとして `u16::MAX`（65,535）行を使用している。これは画面の高さであり入力データのサイズではない — パーサーは任意サイズの入力を処理できる。この設定によりカーソル上書き操作が画面内に収まり、正しく解決される。

## 出力先に応じたフォーマット切り替え

CLI は stdout がターミナルかどうかを検出する（`IsTerminal` トレイト）：

| 出力先                | メソッド                | 結果                          |
|-----------------------|-------------------------|-------------------------------|
| ターミナル (tty)      | `contents_formatted()`  | 色保持、クリーン              |
| ファイル / パイプ / JSON | `contents()`         | プレーンテキスト、エスケープなし |

`ls` や `grep --color=auto` と同じ挙動に準拠している。

## CircleCI Web UI との比較

Docker ビルドステップ（ジョブ #61123、ステップ 106）の場合：

| ソース           | 生データ行数 | vt100 レンダリング後 |
|------------------|-------------|---------------------|
| Private API      | 14,127      | 91                  |
| output_url       | 4,360       | 91                  |
| CircleCI Web UI  | —           | 91                  |

vt100 レンダリングは CircleCI Web UI と同一の出力を再現する。
